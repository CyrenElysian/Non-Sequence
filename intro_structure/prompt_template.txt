You are given a dataset . Each item contains: id (int), scenario (string), unordered_nodes (dictionary), edges (list of directed edges), and script_graph (textual description of the event graph).

The current event graphs have the following problems:
1) There may be spelling errors in scenario and unordered_nodes.
2) Due to previous task limitations, only "sequence" and "and_join" structures are annotated. "and_join" is defined as a structure with weak simultaneity and exchangeable (unordered) branches: branches have no strong dependencies and can be executed in any order or interleaved, but all must be completed before proceeding. Inside each branch, events have a logical order.
3) Some events graphs actually contain "select" or "loop" patterns, but were mis-annotated as "sequence" or "and_join". Example: id 554 where edge 4->1 and node description "Repeat steps 1 and 2" clearly indicate a loop, yet the script_graph is a flat sequence.
**Note:** The input dataset may have been partially corrected by a previous model, but errors can still remain. You should not assume that the edges and script_graph are fully correct. Use your own judgment to identify logical inconsistencies and to decide whether an item should be removed or restructured.

Your task for every item in the dataset:
- Correct spelling errors in scenario and unordered_nodes (only fix obvious misspellings, do not change the intended meaning).
- Identify items where the underlying logic truly contains a select or loop. Remove those items from the final processed dataset because they were wrongly labeled. Record this deletion in the change log.
- For the remaining items, you may optionally introduce at least one select or loop structure (not too many, to keep complexity low) by modifying original nodes and edges if necessary. Requirements:
  - The logic must be natural and not forced. It is acceptable to leave an item unchanged if no select/loop fits.
  - Select branches must be mutually exclusive and significantly different (e.g., choosing between chopsticks and spoon for eating is too trivial).
  - Loops must have a clear exit condition and no infinite loops.
  - For loop specifically: there must be an edge from the node immediately before the loop to entry, an edge from exit to the node immediately after the loop, and an edge from entry to exit . Inside the loop body, the edge sequence must form a path from entry through retry back to entry.
  - For select specifically: each option's last node must have an edge to the same merge point after the select.
  - Remove all old edges that are no longer valid because the script_graph has been restructured. If a previous edge contradicts the new control flow, it must be deleted.
  - After generating the new script_graph and edges, double-check them together: verify that every transition implied by the script_graph (including entering/exiting substructures, loops, and select branches) has a corresponding edge in the edges list, and that no extra edges exist that would create bypasses or dangling nodes.
**Note:** All the above operations should be done together in one pass, not sequentially.

Output Constraints (applies to the processed data file):
- Keep the original JSON structure for each item: id, scenario, unordered_nodes, edges, script_graph.
- General constraints: all structures (including nested ones) must have corresponding nodes and edges; preferably a single entry and single exit for each structure; no dangling nodes or edges; control flow must correctly connect from a substructure’s exit to the parent’s next node; limit nesting depth to keep readability.
- Script_graph format rules:
  - Each structure is enclosed in {}, must have a "type" field (except inside branches_set, which does not have a "type").
  - Arrays [] list items linearly (except for select's options, which are mutually exclusive branches).
  - Arrays can contain other structures.
  - Structure types and their keys:
    - sequence: { "type": "sequence", "script": [] }
    - select: { "type": "select", "options": [] }
    - loop: { "type": "loop", "entry": "...", "retry": [], "exit": "..." }
    - and_join: { "type": "and_join", "branches_set": { "b1": [], "b2": [], ... } }
- Nesting rules:
  - sequence: can contain any structures; internal elements are strictly ordered.
  - select: each option can be any structure (usually a sequence); all options must eventually merge to the same node; options are mutually exclusive with no edges crossing between them.
  - loop: entry is a single node; retry is an array representing the loop body (can contain arbitrary substructures). The last element of retry MUST be the keyword "continue". After executing retry, control flows back to entry. exit is a single node that must have an edge to the node immediately following the loop. There must be no path out of the loop body that bypasses the exit.
  - and_join: branches_set contains named branches (b1, b2, ...). Each branch is an ordered list (can contain substructures). Branches are unordered among themselves but all must complete before the join node. The join node appears after the and_join in the surrounding sequence and is not inside branches_set.
